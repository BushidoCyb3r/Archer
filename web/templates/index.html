      Matched against a finding's source and destination. Wildcards work — *.in-addr.arpa, 185.220.*, *.internal.corp (* = any run, ? = one char); use CIDR (10.0.0.0/8) for IP ranges.       Tuple-scoped view filters — a known-good source → destination : port relationship. Remove a rule and its findings reappear on the next refresh; they were never deleted. Add a rule by right-clicking a finding → "Allow this Relationship".

      Allow this Relationship
      Hides findings on this source → destination : port from the table and the alert bell. Either side can be a single IP, a CIDR range, a domain, or a *.domain wildcard — e.g. source 10.0.0.0/24 → destination 10.0.0.53 port 53 allowlists the whole LAN's resolver relationship in one rule, and destination *.skype.com covers the apex and every name under it (DNS and TI-domain findings carry the domain as their destination). Pure view filter — removing the rule later brings the findings straight back, no re-analysis.

      Beacon
      Spectral rescue runs a frequency-domain analysis (Lomb-Scargle periodogram) on pairs whose statistical timing score was weak. Catches bounded-jitter beacons where adversaries randomize timing around a fixed schedule. Disable if real-world false-positive rate on periodic legitimate traffic (cron jobs, health probes, RSS pollers) outweighs the detection gain.
      Calibration knobs — leave at defaults unless you've measured FP/TP rates on your traffic. See docs/DETECTION_METHODS.md §2 for what each one controls.
      DGA hostname augmentation bumps Beacon / HTTP Beacon scores when the destination's registrable domain looks algorithmically generated (high Shannon entropy + non-English character bigrams). +15 to score, one-step severity upgrade. Built-in CDN allowlist already exempts cloudfront/azure/akamai/etc. Disable if your traffic includes legitimate algorithmic hostnames not covered by the built-in list.
      Calibration knobs — leave at defaults unless you've measured FP/TP rates on your hostnames. See docs/DETECTION_METHODS.md for what each one controls.
      DNS
      Organization Hosts
      The Hosts tab automatically includes RFC 1918 (10/8, 172.16/12, 192.168/16), IPv4 link-local (169.254/16), IPv6 ULA (fc00::/7), and IPv6 link-local (fe80::/10). Add additional CIDRs or single IPs here to surface cloud-hosted servers, owned public blocks, or any non-private address you want monitored as "your host." One per line.
      Threat Intel Lookup APIs
      Per-finding enrichment keys. Used by the Threat Intel Lookup panel on the finding detail view to query each provider on demand. For bulk indicator feeds (MISP, OpenCTI) that drive detection, see the Feeds dialog instead.
      Watch Mode
      By default, only the first watch tick of each UTC day runs the full pipeline (Beacon, HTTP analysis, all detectors); subsequent same-day ticks run a cheaper incremental TI/IOC pass. Turn this on to make every tick a full scan instead — uses more CPU per tick but eliminates the "wait until tomorrow" gap on statistical detectors. Useful during active hunts.
      SIEM forwarding
      Forwards escalated findings as CEF over UDP syslog to your SIEM's syslog input.
      Alerting Thresholds
      Controls when sensor and feed health alarms fire (appearing as badges on the Sensors and Feeds buttons). Quiver's default check-in and rsync schedule is once per hour.

      Audit Log
      The admin-action audit trail is append-only and never edited per entry. A daily sweep removes entries older than the retention window. 0 = keep forever (no automatic deletion) — set a positive value only if a long-running instance needs to bound the table; compliance regimes usually want the full trail.

      Backup
      Download a consistent snapshot of the live SQLite database (findings, notes, audit log, sensor enrollments, allowlist / IOC / suppressions, users). Uses VACUUM INTO so the snapshot captures any in-flight WAL data. The file contains sensitive secrets and credential hashes — handle accordingly. To restore later, stop Archer and replace /data/archer.db with the downloaded file.
      Danger Zone
      Discards every finding in the database and re-analyzes the current log set from scratch. Analyst notes, statuses, and acknowledgements on existing findings are lost. Use after a threshold change when you want a clean baseline.
